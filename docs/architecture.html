Every session is a separate Claude Code terminal running independently. The only shared state is the SQLite database at .claude-swarm/bus.db. Sessions coordinate by reading and writing to this database through the CLI.

Modules

claude_swarm/
+-- bus.py            # SQLite message bus
+-- git.py            # Git operations + push lock
+-- merge.py          # Conflict self-healing
+-- orchestrator.py   # Poll loop + monitoring
+-- cli.py            # Click CLI commands
+-- ao.py             # Quick-start entry point
ModuleResponsibility
bus.pyAll SQLite reads/writes — sessions, messages, reviews, file claims, shared context, sync. The data layer.
git.pyGit operations (add, commit, push, pull, status, diff, log) plus push lock management (acquire, release, check holder).
merge.pyConflict self-healing for non-overlapping changes. Notifies file owners when their claimed files are touched.
orchestrator.pyPoll loop that monitors all sessions. Detects blocked and stale sessions, pending reviews, unread messages. Can auto-approve orchestrator-assigned reviews.
cli.pyThin Click CLI layer over the other modules. All commands delegate to bus.py, git.py, or orchestrator.py.
ao.pyQuick-start entry point. ao setup checks prerequisites, ao start generates templates and launches Claude Code.

Design Decisions

  • SQLite is the only state store. No Redis, no message queues, no external services. A single file (.claude-swarm/bus.db) with WAL mode handles all concurrency. This keeps the tool portable and zero-config.
  • Copy-paste prompts, not sub-agents. The orchestrator prints prompts that you paste into separate Claude Code terminals. Each session is a real, full Claude Code instance — not a constrained sub-agent spawned via the Task tool. This gives each session full autonomy and access to all Claude Code capabilities.
  • Sync is the core primitive. One command (sync) does inbox + context + heartbeat. Sessions run it between every task. This replaces manual inbox checking and ensures no session falls out of date.
  • File claims are advisory. The CLI enforces claims, but there's nothing stopping a session from editing a file it hasn't claimed. The system trusts well-behaved agents and uses claims as a coordination signal, not a hard lock.
  • Reviews gate merges, not commits. Sessions can commit freely. The review gate only blocks merge-ok, which is checked before pushing. This lets sessions work fast and only pause for quality checks at the merge boundary.

Database Schema

6 tables in .claude-swarm/bus.db. SQLite with WAL journal mode and foreign keys enabled.

sessions

Registered sessions with status, workstream, and sync tracking.

CREATE TABLE sessions (
    id          TEXT PRIMARY KEY,
    workstream  TEXT NOT NULL,
    status      TEXT DEFAULT 'running',
    branch      TEXT,
    note        TEXT,
    updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    last_sync_at DATETIME
);
id
Session identifier (e.g. "A", "B", "orchestrator")
workstream
Description (e.g. "frontend", "backend")
status
"running", "blocked", or "done"
branch
Feature branch name (e.g. "feat/frontend")
note
Status message or blocker reason
updated_at
Last heartbeat or status update (UTC)
last_sync_at
Last sync timestamp. Context entries after this are returned on next sync. NULL means first sync returns all context.

messages

Inter-session messages with read/unread tracking.

CREATE TABLE messages (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    from_id     TEXT NOT NULL,
    to_id       TEXT NOT NULL,
    body        TEXT NOT NULL,
    type        TEXT DEFAULT 'message',
    status      TEXT DEFAULT 'unread',
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);
type
"message", "review_request", "review_approved", or "review_rejected"
status
"unread" or "read". Sync marks all unread messages as read.

reviews

Peer review requests with approval/rejection flow.

CREATE TABLE reviews (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    requester   TEXT NOT NULL,
    reviewer    TEXT NOT NULL,
    diff        TEXT,
    status      TEXT DEFAULT 'pending',
    comments    TEXT,
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);
status
"pending" or "approved". Rejection resets to "pending" — the requester must address comments and re-request.

merge-ok returns true only when zero reviews have status != "approved" for the given session.

git_lock

Single-row lock for serializing git pushes.

CREATE TABLE git_lock (
    id          INTEGER PRIMARY KEY CHECK (id = 1),
    held_by     TEXT,
    acquired_at DATETIME
);

Only one row ever exists. held_by is the session ID holding the lock, or NULL. Push acquires the lock, rebases, pushes, then releases. Other sessions wait until the lock is free.

file_claims

Exclusive file ownership per session.

CREATE TABLE file_claims (
    file_path   TEXT PRIMARY KEY,
    session_id  TEXT NOT NULL,
    claimed_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);

One owner per file. Claiming fails if a different session already holds the claim. Same session can re-claim (updates timestamp).

shared_context

Team-wide decisions, interfaces, warnings, conventions, discoveries.

CREATE TABLE shared_context (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id  TEXT NOT NULL,
    category    TEXT NOT NULL,
    body        TEXT NOT NULL,
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);
category
"decision", "interface", "warning", "convention", or "discovery"

Timestamps use Python microsecond precision (not SQLite's second-level CURRENT_TIMESTAMP) to ensure sync filtering works correctly.
